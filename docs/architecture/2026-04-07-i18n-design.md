# i18n Design ŌĆö Tense Master

## Overview

Add internationalization support to the web app. Phase 1: UI text only. Phase 2 (future): exercise content (questions, explanations) in multiple languages.

Supported locales: `ru`, `fr`, `de`, `es`. Default: `ru`.

## Approach

**next-intl** with URL-based locale routing (`/ru/`, `/fr/`, `/de/`, `/es/`).

Rationale:

- Future blog/articles require SEO-friendly locale URLs
- `NextIntlClientProvider` eliminates prop drilling in Client Components ŌĆö the app is predominantly `'use client'`
- Native Next.js 16 dictionary approach requires passing translations as props through the component tree, which is impractical here

## Routing Structure

```
app/
  [lang]/
    layout.tsx       ŌåÉ <html lang={lang}>, <body>, fonts, SerwistProvider, NextIntlClientProvider
    (web)/
      layout.tsx
      page.tsx
      tense-trainer/
      profile/
      settings/
  telegram/          ŌåÉ untouched; locale handled separately when implemented (via Telegram WebApp API)
  api/               ŌåÉ untouched
  globals.css
  sitemap.ts
  not-found.tsx
```

Parameter name: `lang` (Next.js 16 convention, per official docs).

## Proxy (proxy.ts)

Next.js 16 uses `proxy.ts` instead of `middleware.ts`. Located at project root.

Detection order:

1. Locale already in URL (`/ru/...`) ŌåÆ pass through
2. Cookie `NEXT_LOCALE` present ŌåÆ redirect to `/{cookie}/...`
3. `Accept-Language` header ŌåÆ match against `[ru, fr, de, es]`
4. Fallback ŌåÆ `/ru/...`

Excluded paths: `_next`, `api`, `telegram`, static files (`.ico`, `.png`, `.svg`, etc.)

## Layout Setup

`app/[lang]/layout.tsx` is the root HTML layout:

```tsx
export default async function Layout({ children, params }: LayoutProps<'/[lang]'>) {
	const { lang } = await params;
	return (
		<html lang={lang}>
			<body>
				<NextIntlClientProvider locale={lang}>
					<SerwistProvider swUrl='/api/serwist/sw.js'>{children}</SerwistProvider>
				</NextIntlClientProvider>
			</body>
		</html>
	);
}

export function generateStaticParams() {
	return [{ lang: 'ru' }, { lang: 'fr' }, { lang: 'de' }, { lang: 'es' }];
}
```

## Language Selector

Located in `Header` component (visible on all pages).

Behavior:

1. Writes cookie `NEXT_LOCALE={selectedLang}`
2. `router.push('/{selectedLang}' + currentPathnameWithoutLocale)`

## PWA / manifest

`app/manifest.json` (static) ŌåÆ `app/[lang]/manifest.ts` (dynamic Route Handler).

Each locale generates its own manifest:

- `start_url: "/{lang}/"`
- `lang: "{lang}"`

If user installed PWA under `/ru/` and later switches to `fr`: on next launch PWA opens `/ru/`, middleware reads cookie `NEXT_LOCALE=fr` and redirects to `/fr/`. One extra redirect, works correctly.

Service worker scope unchanged ŌĆö Serwist is registered at `/` and covers all locale sub-paths transparently.
